### 1.2 瞬动Git配置

#### 1.2.1 偏好配置

- **界面汉化**：点击右上角头像 → 「Performance」→ 「Localization」→ 「Language」中选择「简体中文」，应用后界面将切换为中文显示。

#### 1.2.2 SSH密钥

**作用**：通过 SSH 密钥实现本地与 Git 服务器的免密通信，提升操作效率。

- **生成密钥对**：打开终端执行以下命令（邮箱替换为个人工作邮箱）：

  ```shell
  ssh-keygen -t rsa -C "your-email@example.com"
  ```

  连续按回车默认存储路径（`~/.ssh/`）和空密码即可。

  

- **获取公钥**：执行命令查看并复制公钥内容：

  ```shell
  cat ~/.ssh/id_rsa.pub
  ```

  示例输出：


## 🏠 二. 实战演示

### 2.1 创建项目

通过平台界面创建新项目，填写项目名称、描述，选择可见性（私有 / 内部 / 公共），初始化时可选择添加`.gitignore`和 README 文件。

- **必填项**：项目名称（建议使用小写字母 + 连字符格式，如`user-management-system`）、项目描述（清晰说明项目用途）
- 可选配置
  - 初始化 README：自动生成项目说明文档
  - 添加`.gitignore`：根据项目类型（如 Java、Python）选择模板，避免提交无关文件
  - 分支保护：创建时可预设 Master 分支保护规则

### 2.2 产品管理

采用任务驱动模式管理产品迭代，包含需求收集、优先级排序、里程碑规划等环节，所有需求需转化为可跟踪的任务。

#### 2.2.1 需求录入规范

- 标题格式：`[模块] 需求描述`（如`[用户中心] 支持第三方账号登录`）
- 内容要素：功能描述、用户场景、验收标准、优先级（P0-P3）、预计工时

#### 2.2.2 迭代规划

- 每个迭代周期（建议 2-4 周）创建里程碑，关联目标版本号（如`v1.2.0`）
- 迭代启动前完成需求评审，确定纳入范围的任务清单



<br/>



### 2.3 代码管理

#### 2.3.1 分支规范

- **Master**：主分支，存放生产环境代码，受保护（禁止直接提交，仅通过合并请求更新）。
- **Dev**：开发分支，集成已完成的功能，作为测试环境代码来源。
- **Feature**：功能分支，命名格式`feature/[需求ID]-xxx`（如`feature/R-123-login`），从 Dev 分支创建，完成后合并回 Dev。
- **Hotfix**：紧急修复分支，命名格式`hotfix/[bugID]-xxx`（如`hotfix/B-456-crash`），从 Master 创建，修复后同时合并到 Master 和 Dev。

#### 2.3.2 代码提交规范

提交信息需清晰描述修改内容，格式如下：

```shell
# 新功能
git commit -m '[feat] 实现用户找回密码功能'

# 修复Bug（关联任务ID）
git commit -m '[fix] #1234 修复移动端验证码显示异常问题'

# 文档更新
git commit -m '[doc] 添加API接口文档说明'

# 测试相关
git commit -m '[test] 补充用户登录模块单元测试'

# 性能优化
git commit -m '[pref] 优化首页加载速度（减少30%请求耗时）'
```

#### 2.3.3 合并请求流程

- 功能完成后，从功能分支向 Dev 分支发起合并请求（MR）

- 至少 1 名团队成员代码审查通过（检查代码规范、逻辑合理性）

- CI 流水线构建通过（自动化测试、代码质量检测）

- 合并前需同步最新 Dev 分支代码，解决冲突

- 合并后删除功能分支，保持仓库分支整洁



<br/>



### 2.4 测试管理

通过测试任务分配、用例管理、缺陷跟踪实现全流程测试覆盖，测试通过的功能方可进入下一环节。

#### 2.4.1 测试流程

- **用例设计**：根据需求创建测试用例，包含用例 ID、标题、前置条件、操作步骤、预期结果

- **任务分配**：将测试用例分配给测试人员，关联对应开发任务

- **执行测试**

  - 测试通过：标记用例状态为「通过」

  - 发现问题：创建 Bug 工单，关联测试用例和功能分支

- **回归验证**：开发修复后，测试人员验证 Bug 是否解决，确认后关闭工单



#### 2.4.2 Bug 管理规范

- 标题格式：`[模块] 问题描述`（如`[支付页] 点击提交按钮无响应`）

- 内容要素：复现步骤、实际结果、预期结果、截图 / 录屏、环境信息（浏览器 / 设备型号）

- 优先级划分：

  - P0：阻断性问题（如程序崩溃、功能完全不可用）

  - P1：严重问题（核心功能异常，无替代方案）

  - P2：一般问题（功能可用但体验不佳）

  - P3：建议性问题（优化项）

### 2.5 任务看板

使用看板可视化任务状态（待处理、进行中、已完成等），实时跟踪项目进度，支持拖拽调整任务状态。

![771](asset/771.png)

#### 2.5.1 看板列配置

- 「需求池」：未规划的需求任务
- 「待处理」：已纳入迭代，等待开发的任务
- 「进行中」：开发人员正在处理的任务（需关联功能分支）
- 「待审核」：开发完成，等待代码审查的任务
- 「测试中」：已合并到 Dev，等待测试验证的任务
- 「已完成」：测试通过，可纳入发布的任务



#### 2.5.2 任务流转规则

- 任务创建后默认进入「需求池」，规划后移入「待处理」
- 领取任务后从「待处理」移至「进行中」，并更新负责人和预计完成时间
- 开发完成后提交合并请求，任务移至「待审核」
- 代码审查通过后移至「测试中」，关联测试用例
- 测试通过后移至「已完成」，关闭任务

### 2.6 高级用法

- **CI/CD 自动化**：配置流水线实现代码提交后自动构建、测试、部署，减少人工操作，提升交付效率。
- **分支保护**：对 Master、Dev 等核心分支设置保护规则，要求代码审查通过后才能合并。
- **协作冲突解决**：使用`git pull --rebase`同步远程更新，通过`git mergetool`解决代码冲突。

## 👨‍🏭 三. 工作要求

1. ❤️ **实时关注**：每日高频查看个人 TODO、任务和 Bug 列表，确保及时响应。
2. ❤️ **留痕原则**：所有操作（代码提交、任务变更、沟通记录等）需留存记录，便于追溯。
3. ❤️ **任务化管理**：工作内容必须拆解为具体任务，明确目标、负责人和截止时间。
