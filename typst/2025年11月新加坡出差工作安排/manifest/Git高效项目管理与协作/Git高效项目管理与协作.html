一. Git配置

1.1 客户端下载

  • 官方客户端： Git 官方下载 (推荐专业用户，轻量高效)
  • 图形化工具： TortoiseGit (适合新手，可视化操作界面)

1.2 瞬动Git配置

Git地址：https://git.shundong.xyz

1.2.1 偏好配置

  • 界面汉化：点击右上角头像 → 「Performance」→ 「Localization」→ 「Language」中选择「简体中文」，应用后界面将切换为中文显示。
  • 首页定制：通过「头像 → Performance → Behavior → Homepage」设置默认首页。推荐选择「Member Projects」（团队项目）或「Starred Projects」（星标项目），快速访问常用内容。

1.2.2 SSH密钥

作用：通过 SSH 密钥实现本地与 Git 服务器的免密通信，提升操作效率。

生成密钥对：

打开终端执行以下命令（邮箱替换为个人工作邮箱）：

ssh-keygen -t rsa -C "your-email@example.com"

连续按回车默认存储路径（~/.ssh/）和空密码即可。

获取公钥：

执行命令查看并复制公钥内容：

cat ~/.ssh/id_rsa.pub

示例输出：

配置到 Gitlab：

登录瞬动 Git 平台，进入个人设置 → 「SSH 密钥」→ 粘贴公钥内容并保存。

1.2.3 终端个性化

添加个性化配置&常用技巧，参考 https://blog.csdn.net/Gusand/article/details/126214596#t5

常用别名速查表（简化命令输入）：

别名 对应命令 说明
git st git status 查看工作区文件状态
git br git branch 列出 / 创建 / 删除分支
git ci git commit 提交暂存区到本地仓库
git ss git add . 暂存所有工作区修改

二. 实战演示

2.1 创建项目

通过平台界面创建新项目，填写项目名称、描述，选择可见性（私有 / 内部 / 公共），初始化时可选择添加.gitignore和 README 文件。

  • 必填项：项目名称（建议使用小写字母 + 连字符格式，如user-management-system）、项目描述（清晰说明项目用途）
  • 可选配置
    • 初始化 README：自动生成项目说明文档
    • 添加.gitignore：根据项目类型（如 Java、Python）选择模板，避免提交无关文件
    • 分支保护：创建时可预设 Master 分支保护规则

2.2 产品管理

采用任务驱动模式管理产品迭代，包含需求收集、优先级排序、里程碑规划等环节，所有需求需转化为可跟踪的任务。

2.2.1 需求录入规范

  • 标题格式：[模块] 需求描述（如[用户中心] 支持第三方账号登录）
  • 内容要素：功能描述、用户场景、验收标准、优先级（P0-P3）、预计工时

2.2.2 迭代规划

  • 每个迭代周期（建议 2-4 周）创建里程碑，关联目标版本号（如v1.2.0）
  • 迭代启动前完成需求评审，确定纳入范围的任务清单

2.3 代码管理

2.3.1 分支规范

Master

主分支，存放生产环境代码，受保护（禁止直接提交，仅通过合并请求更新）。

Dev

开发分支，集成已完成的功能，作为测试环境代码来源。

Feature

功能分支，命名格式feature/[需求ID]-xxx，从 Dev 分支创建，完成后合并回 Dev。

Hotfix

紧急修复分支，命名格式hotfix/[bugID]-xxx，从 Master 创建，修复后同时合并到 Master 和 Dev。

2.3.2 代码提交规范

提交信息需清晰描述修改内容，格式如下：

# 新功能

git commit -m '[feat] 实现用户找回密码功能'

# 修复Bug（关联任务ID）

git commit -m '[fix] #1234 修复移动端验证码显示异常问题'

# 文档更新

git commit -m '[doc] 添加API接口文档说明'

# 性能优化

git commit -m '[pref] 优化首页加载速度（减少30%请求耗时）'

2.3.3 合并请求流程

  1. 功能完成后，从功能分支向 Dev 分支发起合并请求（MR）
  2. 至少 1 名团队成员代码审查通过（检查代码规范、逻辑合理性）
  3. CI 流水线构建通过（自动化测试、代码质量检测）
  4. 合并前需同步最新 Dev 分支代码，解决冲突
  5. 合并后删除功能分支，保持仓库分支整洁

2.4 测试管理

通过测试任务分配、用例管理、缺陷跟踪实现全流程测试覆盖，测试通过的功能方可进入下一环节。

2.4.1 测试流程

  • 1
    用例设计：根据需求创建测试用例，包含用例 ID、标题、前置条件、操作步骤、预期结果
  • 2
    任务分配：将测试用例分配给测试人员，关联对应开发任务
  • 3
    执行测试
    • 测试通过：标记用例状态为「通过」
    • 发现问题：创建 Bug 工单，关联测试用例和功能分支
  • 4
    回归验证：开发修复后，测试人员验证 Bug 是否解决，确认后关闭工单

2.4.2 Bug 管理规范

  • 标题格式：[模块] 问题描述（如[支付页] 点击提交按钮无响应）
  • 内容要素：复现步骤、实际结果、预期结果、截图 / 录屏、环境信息（浏览器 / 设备型号）
  • 优先级划分：
    P0

    阻断性问题

    P1

    严重问题

    P2

    一般问题

    P3

    优化项

2.5 任务看板

使用看板可视化任务状态（待处理、进行中、已完成等），实时跟踪项目进度，支持拖拽调整任务状态。

2.5.1 看板列配置

需求池

未规划的需求任务

待处理

已纳入迭代，等待开发

进行中

开发人员正在处理

待审核

等待代码审查

测试中

等待测试验证

已完成

可纳入发布

2.5.2 任务流转规则

需求池
待处理
进行中
待审核
测试中
已完成
  • 任务创建后默认进入「需求池」，规划后移入「待处理」
  • 领取任务后从「待处理」移至「进行中」，并更新负责人和预计完成时间
  • 开发完成后提交合并请求，任务移至「待审核」
  • 代码审查通过后移至「测试中」，关联测试用例
  • 测试通过后移至「已完成」，关闭任务

2.6 高级用法

CI/CD 自动化

配置流水线实现代码提交后自动构建、测试、部署，减少人工操作，提升交付效率。

分支保护

对 Master、Dev 等核心分支设置保护规则，要求代码审查通过后才能合并。

协作冲突解决

使用git pull --rebase同步远程更新，通过git mergetool解决代码冲突。

三. 工作要求

实时关注

每日高频查看个人 TODO、任务和 Bug 列表，确保及时响应。

留痕原则

所有操作（代码提交、任务变更、沟通记录等）需留存记录，便于追溯。

任务化管理

工作内容必须拆解为具体任务，明确目标、负责人和截止时间。

遵循以上要求，可有效提升团队协作效率，保证项目质量和交付进度。
